🔄 CI/CD Pipeline — GitHub Actions

Pipeline automatizado alineado con los scripts de package.json: lint → typecheck → test → build → docker.

Scripts de package.json que alimentan el Pipeline

Cada etapa del CI ejecuta uno o más scripts definidos en la raíz del monorepo:

Etapa CIScriptComando RealBloquea?
Lintingnpm run linteslint . --ext .ts,.tsx✅ Sí
Type Checknpm run typechecktsc --noEmit✅ Sí
Unit Testsnpm run test:unitjest --coverage --testPathIgnorePatterns=tests/integration✅ Sí (cobertura ≥80%)
Integration Testsnpm run test:integrationjest --config jest.integration.config.ts✅ Sí
Build Servernpm run build:servercd src/server && tsc✅ Sí
Build Webnpm run build:webcd src/web && next build✅ Sí
Docker Buildnpm run docker:builddocker compose -f docker-compose.yml build✅ Sí

Workflow por Branch (Recomendado)

BranchTriggerEtapas ejecutadas
feature/*Push / PRLint → TypeCheck → Unit Tests
developMerge a developTodo lo anterior + Integration Tests + Docker Build + Container Scan
release/*Push tag semverTodas las etapas + deploy a staging + smoke tests
mainMerge a mainTodo + deploy producción + DB migrations

Ejemplo de GitHub Actions Workflow

# .github/workflows/ci.yml
name: CI Pipeline
on:
  push:
    branches: [main, develop, 'feature/*']
  pull_request:
    branches: [main, develop]

jobs:
  lint-typecheck:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '22'
          cache: 'npm'
      - run: npm ci
      - run: npm run lint
      - run: npm run typecheck

  test-unit:
    needs: lint-typecheck
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '22'
          cache: 'npm'
      - run: npm ci
      - run: npm run test:unit

  test-integration:
    needs: lint-typecheck
    if: github.ref == 'refs/heads/develop'
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:17-alpine
        env:
          POSTGRES_DB: crux_mail
          POSTGRES_USER: crux_user
          POSTGRES_PASSWORD: test_password
        ports: ['5432:5432']
      redis:
        image: redis:7-alpine
        ports: ['6379:6379']
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '22'
          cache: 'npm'
      - run: npm ci
      - run: npm run test:integration
        env:
          TEST_DB_URL: postgresql://crux_user:test_password@localhost:5432/crux_mail
          REDIS_URL: redis://localhost:6379

  docker-build:
    needs: [test-unit, test-integration]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: docker compose -f docker-compose.yml build

  security-scan:
    needs: docker-build
    runs-on: ubuntu-latest
    steps:
      - uses: aquasecurity/trivy-action@master
        with:
          image-ref: 'crux-webmail:latest'
          exit-code: '1'
          severity: 'CRITICAL,HIGH'

Seguridad en el Pipeline

  • Secret Scanning: Gitleaks ejecuta en cada commit para detectar credenciales hardcodeadas.
  • Dependency Audit: npm audit + Dependabot automatizado para alertas de vulnerabilidades.
  • Container Scanning: Trivy escanea imágenes Docker; CRITICAL y HIGH bloquean merge.
  • Base Images: Alpine / distroless minimizadas, con versiones específicas (p.ej. postgres:17-alpine, redis:7-alpine).
  • Infraestructura inmutable: Los secrets del docker-compose se leen desde archivos, no desde variables de entorno.

Variables de Entorno en CI

Los secretos se inyectan vía GitHub Encrypted Secrets. Nunca aparecen en logs:

# Ejemplo de inyección en GitHub Actions:
env:
  NODE_ENV: test
  TEST_DB_URL: ${{ secrets.STAGING_DB_URL }}
  JWT_SECRET: ${{ secrets.JWT_SECRET }}
  REDIS_URL: redis://localhost:6379
  # Docker registry auth
  REGISTRY_URL: ghcr.io/${{ github.repository_owner }}
  REGISTRY_TOKEN: ${{ secrets.GHCR_TOKEN }}
