📈 Escalabilidad

Estrategias de escalamiento horizontal y vertical del stack, con límites por servicio y auto-scaling en Kubernetes.

Arquitectura de Escalamiento

Crux-Webmail está diseñado para escalar servicios stateless horizontalmente, mientras que los servicios stateful se escalan verticalmente o con réplicas específicas:

Escalamiento Horizontal de Fastify

Como Fastify es stateless (sesiones en Redis, caché en Redis), se puede replicar libremente:

# Escalar Fastify a 4 instancias
docker compose up -d --scale server=4

# Verificar que todas están healthy
docker compose ps server

# Nginx round-robin configura:
upstream fastify_backend {
    server crux-server-1:3000;
    server crux-server-2:3000;
    server crux-server-3:3000;
    server crux-server-4:3000;
}

Redis Sentinel para HA

Para alta disponibilidad de sesiones y rate-limit:

# Configurar Redis Sentinel (3 nodos mínimo)
sentinel monitor redis-master 172.21.0.15 6379 2
sentinel down-after-milliseconds redis-master 5000
sentinel failover-timeout redis-master 15000
sentinel parallel-syncs redis-master 1

PostgreSQL Read Replicas

Para queries pesadas (search, reporting, dashboards):

# Read replica con streaming replication
# docker-compose.yml add:

postgres-replica:
  image: postgres:17-alpine
  networks:
    data-network:
  environment:
    POSTGRES_USER: crux_user
    POSTGRES_PASSWORD_FILE: /run/secrets/postgres_password
  command: >
    postgres -c hot_standby=on
             -c wal_level=replica
             -c max_wal_senders=3
             -c hot_standby_feedback=on
  depends_on:
    postgres:
      condition: service_healthy

Auto-scaling con Kubernetes

Para producción enterprise, migrar a K8s con HPA:

# HorizontalPodAutoscaler para Fastify
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: fastify-server-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: fastify-server
  minReplicas: 2
  maxReplicas: 20
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 80
    - type: Pods
      pods:
        metric:
          name: http_requests_per_second
        target:
          type: AverageValue
          averageValue: 1000

Consideraciones Zero-Trust al Escalar

  • Cada nueva réplica Fastify debe tener acceso a Redis (sesiones) y PostgreSQL
  • mTLS entre servicios debe configurarse en cada nuevo contenedor
  • Nginx upstream debe incluir health checks activos
  • Rate limiting de @fastify/rate-limit es compartido via Redis — no se duplica
  • BullMQ workers pueden escalar independientemente de Fastify servers
