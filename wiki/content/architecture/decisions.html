📋 ADRs — Decisiones Arquitectónicas

Registro de decisiones técnicas con justificación y trade-offs.

ADR-001: Fastify sobre Express.js

Status✅ Aceptado
MotivaciónFastify ofrece ~3x throughput, JSON schema validation nativa, plugin system aislado, y menor overhead de memoria
AlternativasExpress (mature pero más lento), Koa (minimalista pero sin schema), Hapi (opinionado pero mayor bundle)
ConsecuenciasMayor performance en producción, validación automática, mejor DX con TypeScript

ADR-002: Next.js 14 con App Router

Status✅ Aceptado
MotivaciónSSR para SEO del login/landing, Server Components para reducir client JS, RSC para data fetching optimizado
AlternativasSPA vanilla (pero sin SSR), Remix (similares pero más nueva, menos ecosystem), Vue Nuxt (stack differente)
ConsecuenciasMejor LCP, menor TTFB, Server Actions para mutations sin API endpoints adicionales

ADR-003: PostgreSQL + Sequelize sobre MongoDB

Status✅ Aceptado
MotivaciónRelaciones complejas (users↔mailboxes↔emails↔folders), ACID transactions para envío,成熟的 backup tooling, ACID para email routing
AlternativasMongoDB (flexible schema pero sin transactions robustas en ese momento), MySQL (menos tipos de datos, peor JSON support)
ConsecuenciasEsquema estricto documentado en Sequelize models, migraciones versionadas, backups pg_dump

ADR-004: Docker Compose sobre Kubernetes

Status✅ Aceptado
MotivaciónSingle-node deployment típico, ~10 contenedores max, simplicidad operativa, menor curva de aprendizaje
AlternativasK8s (overkill para single-node), systemd services (menos aislación), VMs (más overhead)
ConsecuenciasEscalabilidad horizontal limitada a single-host. Migración a K8s posible via docker-compose→helm

ADR-005: WebCrypto para E2E Encryption

Status✅ Aceptado
MotivaciónCifrado del lado del cliente antes de enviar datos al server. AES-GCM + RSA-OAEP. Browser native, sin dependencias externas
AlternativasOpenPGP.js (más pesado, key management complejo), libsodium-wrappers (no native browser)
ConsecuenciasServer nunca ve datos en plaintext. Key rotation manual. Fallback para browsers sin WebCrypto necesario

ADR-006: JMAP Bridge sobre IMAP nativo en frontend

Status✅ Aceptado
MotivaciónJSON protocol moderno (RFC 8620), batching, posChangeKey para sync eficiente, no necesita IMAP client library en browser
AlternativasIMAP directo (websockets + imap-client), IMAP over SMTP (no estándar)
ConsecuenciasBridge adicional pero desacoplado. Frontend habla solo HTTP/JSON.
